## 1. 서비스 정체성

**AI Champion**은 AX(AI Transformation) 도입 초기 단계에 있는 "개인 중간관리자"를 위한 코칭 어시스턴트 서비스다.

전사 단위 컨설팅이 아니라, 관리자 한 사람의 의지만으로 자기 팀 단위에서 AI 활용을 시작하고, 실험하고, 성과를 검증할 수 있도록 돕는다.

| | 기존 AX 지원 서비스 | AI Champion |
|---|---|---|
| 주체 | B2B 엔터프라이즈 | 관리자 한 사람의 의지만으로 |
| 속도 | 보안 등 정책과 이해관계로 제한 | Agile |
| 단계 | AI 도입 중심 | 조직 내 확산과 정착 |
| 목표 | 범용 도구, 교육 | AI로 업무 재설계, 팀 차원 활용을 측정·코칭·평가 |

### 사용자 특징

- AI/AX 관련 전문 지식이 부족한 비전문가일 가능성이 높음
- 팀 규모 3~15명 내외의 중간관리자 (사원~부장급)
- 회사 차원의 AI 가이드라인이나 예산 지원이 없는 경우가 많음
- "AI를 써야 하는 건 알겠는데 어디서부터 시작해야 할지 모르는" 상태

### AX 성공전략 (AI Champion이 추구하는 방향)

- **현업 중심 문제 정의가 기술보다 우선**: 사용자가 "어떤 업무가 반복되는가, 어디서 비효율이 발생하는가"를 먼저 파악하고, AI를 해결 수단으로 인식하게 함. Best Practice를 찾아 우리 조직의 일에 적용·확산.
- **직무 기반의 직관적 BP**: 구성원이 바로 이해할 수 있는 직무 단위 AI 우수 사례 제공.
- **프로세스 재설계 관점**: 단순 자동화가 아니라 업무 프로세스 재설계. 직무 단위가 아닌 스킬 단위 인력 재배치까지 검토.
- **AI 만능주의 경계**: 업무에 적합한 AI tool·전략을 제시하되, 생성형 AI 사용이 오히려 비용/비효율을 유발하는 업무는 과감히 pivot 제안.

## 2. 준수 정책

세부 기능/정책이 새로 합의될 때마다 이 섹션에 계속 추가·수정한다.

### 2.1 쉬운 언어 사용

기술 용어(RAG, 파인튜닝, 임베딩 등)를 사용자에게 노출하지 않는다. 대신 비유와 일상어로 풀어 설명한다.

- 예: "우리 회사 자료를 참고해서 답하는 방식" (RAG 대신), "AI에게 말투를 가르치는 것" (파인튜닝 대신)

### 2.2 Layer 구분

업무를 추천하거나 자동화 포인트를 제안할 때는 반드시 아래 3단계 중 어디에 해당하는지 명시하고, 각 Layer의 원칙을 지킨다.

| Layer | 정의 | 원칙 | 예시 |
|---|---|---|---|
| Layer 1 | 반복·정형 업무 | AI가 처리 결과를 바로 활용 가능한 수준으로 생성 | 회의록 정리, 데이터 취합, 정형 보고서 초안 |
| Layer 2 | 비정형·판단 필요 업무 | AI는 초안/보조자료만 생성. 사람이 맥락을 채워야 하는 지점을 명확히 표시 | 팀원 피드백 문구, 일정 조율, 1차 분석 |
| Layer 3 | 전략·의사결정 업무 | AI는 정보 정리와 선택지 제시까지만. "최종 판단은 사용자의 몫"임을 항상 명시 | 예산 배분, 인사 평가, 조직 개편 |

Layer 3 업무에 대해 AI가 결론이나 확정적 판단을 대신 내리는 것처럼 보이는 표현은 절대 사용하지 않는다.

### 2.3 AI 만능주의 경계

사용자가 AI로 해결하기 부적합한 문제(예: 팀원 간 신뢰 문제, 조직문화 이슈)를 자동화하려 할 경우 이를 지적하고, AI가 도울 수 있는 부분과 사람이 직접 해야 하는 부분을 구분해서 안내한다.

### 2.4 실행 가능성 최우선

추상적 조언보다 "지금 바로 복사해서 쓸 수 있는" 프롬프트, 체크리스트, 구체적 다음 행동을 우선 제시한다. 일반 컨설팅 리포트 식의 뭉뚱그린 전략 언어를 쓰지 않는다.

### 2.5 작게 시작해서 검증

전사 도입이나 큰 예산이 필요한 제안은 하지 않는다. 항상 "이번 주에 팀원 1~2명과 시도해볼 수 있는 것"부터 제안한다.

### 2.6 절대 하지 않는 것

- 사용자의 회사 기밀 정보나 개인정보를 요구하거나 저장 제안
- 검증되지 않은 통계·수치를 확정적으로 제시 (출처가 있는 경우에만 인용)
- Layer 3 업무에 대한 최종 의사결정 대행
- 팀원 해고, 평가 불이익 등 인사상 불이익과 직결되는 업무의 자동화 제안

## 3. 전체 플로우

```
업무 파악 → AI 활용 현황 파악 → AI Opportunity 발견 → 팀 자산화 (Loop)
```

이 플로우를 아래 6개 기능이 순서대로 구현하며, 5→6 이후 다시 진단/로드맵 갱신으로 순환(Loop)한다.

이 큰 흐름 안에, 4.4↔4.5 사이에는 **계획 → 실행 → 피드백 → 자산화**로 더 촘촘하게 도는 하위 루프가 있다. 4.4(계획/로드맵)가 만든 task를 4.5(실행/트래킹)가 실행·피드백을 수집하고, 그 결과가 4.5의 자산화 저장소에 쌓여 다음 루프의 4.4에 AI Opportunity 후보로 다시 들어간다. 4.3(BP 리서치 엔진)은 이 내부 루프와 별개로, 매 루프마다 4.4에 **외부** 트렌드 컨텍스트를 공급하는 역할이다.

## 4. 기능 Skeleton (6개 고정)

기능 개수는 6개로 고정한다. 신규 세부 기능은 아래 항목 내부의 하위 요소로 편입하며, 기능 자체를 추가/삭제하지 않는다.

### 4.1 온보딩 인터뷰

- **목적**: 사용자(중간관리자)와 팀의 기본 맥락 파악의 출발점.
- **질문 설계 원칙**: "자동화할 업무를 입력하세요" 처럼 추상적으로 물으면 비전문가는 답을 못 한다. 대신 "하루 업무를 시간순으로 알려주세요" 처럼 구체적으로 답하기 쉬운 질문으로 유도한다.
- **하위 요소** (4개 파트로 나눠 진행):
  - 기본 정보: 업종, 팀 규모, 담당 업무 카테고리
  - AI 활용 수준: 4단계(안 씀 / 궁금해서 써봄 / 가끔 필요할 때 씀 / 업무에 적극 활용)
  - 조직 환경: 회사 AI 가이드라인 유무, 사내 지정 AI 도구, 외부 AI 사용 가능 여부, 팀원 간 AI 활용 수준 편차
  - 반복 업무 상세 (핵심): 하루 업무를 시간순으로 나열 → 언급된 업무마다 빈도/정형성(매번 비슷한지)/평균 소요시간/민감정보 포함 여부/기존 처리 방식을 추가 확인
  - (역할 재분배용, 선택) 팀원별 강점 영역·AI 활용 편안함 정도·현재 업무 부담도를 관리자의 주관적 판단으로 간단 태깅
- **입력**: 사용자 답변
- **출력**: 팀 프로필 + 반복 업무 리스트 (성숙도 진단·로드맵 생성의 입력값)
- **정책 연계**: 회사 기밀/개인정보 요구·저장 금지 (2.6) — 팀원 태깅은 이름 대신 익명 식별자 사용 권장
- **오픈 이슈**: 정확한 질문 문구, 인터뷰 형식(대화형 vs 설문형) 세부 설계 미정

### 4.2 AX 성숙도 진단 및 목표 설정

- **목적**: 시작점 파악 + 목표 설정
- **하위 요소**:
  - 시작점 파악: 현재 AX 성숙도를 5개 축으로 진단해 레이더 차트로 출력
    - 5개 축: **전략 명확성 / 도구 활용도 / 팀 수용력 / 데이터 접근성 / 평가 체계**
    - 축별 점수 + 간단한 해석 코멘트(예: "도구 활용도 낮음 → 허용된 AI 도구가 없음") 제공
    - (선택) 공신력 있는 출처가 있는 외부 AX 리포트 통계와 비교한 벤치마크 코멘트 제공 가능 — 반드시 출처 명시 (2.6)
  - 목표 설정: (문제 정의) → 우리 조직의 문제를 AI로 해결 → 실제 가치 창출
- **입력**: 온보딩 인터뷰 결과 (팀 프로필, 반복 업무 리스트)
- **출력**: 성숙도 진단 결과(레이더 차트 + 축별 우선순위) + 목표 정의서
- **정책 연계**: 미검증 통계 확정적 제시 금지 (2.6) — 벤치마크 코멘트는 출처 있는 경우에만
- **오픈 이슈**: 축별 점수 산출 기준(문항·배점), 우선순위 도출 로직 미정

### 4.3 BP 리서치 엔진

- **정체성 (기원)**: "조직 내에서 업무에 AI를 잘 활용한 사람이 동료들에게 노하우를 공유한다"는 아이디어에서 출발한 기능이지만, 실제 산출물은 사용자에게 직접 보여주는 화면이 아니라 **4.4(로드맵 생성)가 참고하는 백엔드 리서치 레이어**다.
- **목적**: 1~2번에서 정한 목표를 달성할 구체적 방법을 4.4가 생성할 때, 그 근거가 될 외부 컨텍스트를 조사해서 넘겨준다.
- **리서치 대상 (웹서치 기반)**: AX 트렌드, 논문/연구 자료, frontier한 개인의 AI 활용법 등 — 특정 회사/리포트에 국한하지 않고 폭넓게 조사
- **입력**: 2번의 목표 정의서 (목표 텍스트 + 조직 제약: 허용 도구, 연동 시스템, 보안 수준 등) — **목표 단위**로 조사한다. 아직 쪼개지지 않은 task 단위로 조사하지 않는다 (task breakdown은 4.4의 역할)
- **사용자 노출 여부**: **없음.** 사례 카드나 리스트 형태로 사용자에게 직접 보여주지 않는다. 4.4가 이 컨텍스트를 참고해 로드맵/근거를 생성하고, 사용자에게는 4.4의 결과물(쉬운 언어로 풀어쓴 로드맵)만 노출된다.
- **출력**: 4.4에 전달하는 리서치 컨텍스트 (목표와 관련된 트렌드/사례/방법론 요약). 스키마는 스프린트 문서(`SPRINT1_CONTRACT.md`) 참고
- **정책 연계**: 출처 있는 경우만 인용(2.6) — 웹서치 결과를 4.4가 사용자에게 풀어 설명할 때 출처 추적이 가능해야 함
- **3↔4 경계 (확정)**: 3번은 검색·조사만 담당(생성·판단 없음), 4번은 3번의 컨텍스트를 받아 foundation 모델로 생성·판단(검색 없음)한다. 3번의 리서치는 실시간 웹서치 기반으로 하며, 사전 구축된 문서 corpus 매칭 방식은 채택하지 않는다.
- **오픈 이슈**: 리서치 갱신 주기, 소스 신뢰도 기준. 세부 구현은 `SPRINT1_FEATURE3_BP_RESEARCH.md` 참고
  - 자산화 저장소(팀 내부 실행 기록)는 이 기능이 아니라 4.5 소관으로 분리됨 — 이 기능은 **외부** 리서치만 다룬다

### 4.4 맞춤 로드맵 + 평가 지표 생성

- **목적**: 목표를 실행 가능한 task로 분해하고, 수치화·해석 가능한 성과 지표 설계
- **처리 순서 (하위 요소)**:
  1. **AI 적합성 판정** (신규 — "AI 만능주의 경계" 정책의 실행 장치): 온보딩에서 수집한 반복 업무를 Layer 분류 이전에 먼저 "이거 AI로 풀 문제가 맞는가"부터 판정한다.
     - 판정 매트릭스: **빈도(자주 / 가끔) × 정형성(정형적 / 비정형)**
       - 자주+정형 → 규칙기반 자동화(엑셀 함수/템플릿)를 추천하고 생성형 AI는 과함을 안내 (Pivot)
       - 자주+비정형 → 생성형 AI 최적 영역, Layer 분류로 진행
       - 가끔+정형 → 자동화 투자 대비 효율 낮음, 현행 유지 추천 (Pivot)
       - 가끔+비정형 → 케이스바이케이스, 우선순위 낮게 참고자료 정도로만 활용
     - 게이트(판정을 강등시키는 조건): 민감정보 포함 + 회사 가이드라인 없음 → "보류 + 경고 카드"로 강등. 이미 기존 대안으로 충분히 처리되고 있음 → "현행 유지"로 강등
     - 부적합 판정 시 **Pivot 카드** 발행: 이유 + 대안(기존 도구/수작업 유지)을 함께 안내. 이 결정은 4.5의 자산화 저장소에 Pivot 기록으로 남는다
  2. Layer 1/2/3 분류 (적합 판정된 업무만): 각 task에 Layer 표시, task 필드는 **난이도 / 예상 소요시간 / 기대 효과 / 필요 도구 / 실패 요인**을 포함
  3. 목표 → 로드맵 task로 break-down (주 단위 타임라인)
  4. **팀 내 역할 재분배 제안** (신규, 경량 스코프): task 담당자를 정할 때, 온보딩에서 태깅한 팀원 강점/AI 활용 편안함/업무 부담도를 참고해 재분배 제안 카드를 생성
     - 스코프는 **팀 내부 재분배로 한정** — 팀 밖 이동, 채용/충원 제안은 하지 않는다
     - 인사평가·급여 등 인사 데이터는 요구하지 않으며, 관리자의 주관적 관찰만으로 작동
     - 모든 제안 카드에 "실제 배분은 팀장님이 판단해주세요" 고정 문구 포함 (Layer 3 원칙과 동일 — 실행 권한 없음)
  5. 성과 지표 설계 (업무시간 단축/생산성 증가 등, 주간 progress 측정용) — 이후 Bottom-up 설득용 리포트 작성의 기반
- **입력**: 목표 정의서(4.2) + BP 리서치 엔진의 컨텍스트(4.3) + 온보딩의 반복 업무·팀원 태깅 데이터(4.1) + (2회차 이후) 4.5 자산화 저장소에 쌓인 이전 실행 기록·Pivot 기록
- **출력**: AI 적합성 판정 결과(Pivot 카드 포함) + 로드맵(task 목록, Layer 표시) + 역할 재분배 제안 카드 + 평가 지표 세트
- **정책 연계**: AI 만능주의 경계(2.3) — 적합성 판정 단계로 구체화. 실행 가능성 최우선(2.4) — task는 "이번 주 바로 시도 가능한" 수준으로 구체화. 작게 시작(2.5). Layer 구분(2.2). 인사 불이익 자동화 금지(2.6) — 역할 재분배는 팀 내부·비인사 데이터로 한정
- **오픈 이슈**: 지표 산식(시간 단축 측정 방법), task granularity 기준, 적합성 판정 매트릭스의 구체 임계값(예: "자주"의 기준 빈도) 미정. 세부 구현은 `SPRINT1_FEATURE4_ROADMAP_GENERATOR.md` 참고

### 4.5 로드맵 실행 트래킹 / 주간 코칭 및 평가

- **목적**: 로드맵 실행 현황을 추적하고 주간 단위로 코칭·평가. 동시에 "1회성 진단"이 아니라 **지속적으로 새 AI Opportunity를 발견하는 루프**를 운영한다 (3장 전체 플로우의 반복 구조를 구현하는 지점)
- **하위 요소**:
  - 로드맵 진행률 체크: task별 실행 여부·소요시간 확인
  - **지속적 발견 루프** (신규):
    - 주간 체크인 (가볍게, 3문항): "이번 주 귀찮았던 순간이 있었나요?" → "대신해줬으면 싶었나요, 원래 해야 하는 일이었나요?" → "AI 도구 써봤다면 무엇을, 어땠는지 한 줄로"
    - 월간 업무 패턴 스캔 (깊게, 필요 시): 흔한 업무 패턴 6종 카탈로그(정보탐색형/반복소통형/정기문서형/판단비교형/기획초안형/반복설명형)를 예시와 함께 제시해 스스로 못 찾는 기회를 유도
    - (선택) 팀원 익명 페인포인트 수집: 팀원에게 짧은 익명 질문을 보내 관리자 시야 밖의 이슈 파악, 여러 명이 겹쳐 언급한 항목은 우선순위를 높게 반영
    - 새로 발견된 후보는 4.4의 AI 적합성 판정부터 다시 통과시킨다 (새 파이프라인을 따로 만들지 않음)
  - **막힘 감지**: 로드맵 진행률이 정체되면 자동으로 "왜 안 쓰게 됐는지" 체크인 질문을 보내고, 답변에 따라 로드맵을 재조정한다
  - **활용 대시보드**: 개인별 활용 현황(팀원 본인이 보는 화면)과 팀 전체 활용 현황(관리자가 보는 화면)을 제공. 관리자 뷰는 "누구부터 도와줘야 할지" 판단을 돕는 코칭 보조 목적으로만 사용하며, 인사평가와는 명시적으로 분리한다
  - **팀 자산화 저장소** (4.3에서 이관 — 내부 실행 기록 전담): "계획(4.4) → 실행(4.5) → 피드백(4.5) → 자산화" 루프를 닫는 저장소. 4.3(BP 리서치 엔진)이 다루는 **외부** 트렌드/사례 조사와는 분리된, **팀 내부에서 실제로 실행한 내용의 기록**이다.
    - 저장 대상: 실행한 task와 결과(성공/실패), 재사용 가능한 프롬프트, **Pivot 기록**("이 업무는 AI로 안 풀린다"고 판단한 이유), 체크인·패턴 스캔에서 나온 미착수 아이디어
    - 다음 루프 반영: 이 기록은 다음 사이클의 4.4(로드맵 생성)에 **AI Opportunity 후보**로 그대로 입력되어, 매번 새로 논의하지 않고도 "이미 검증된 것/이미 아니라고 판단한 것"을 이어받는다
- **입력**: 로드맵/지표(4.4) + 사용자의 주간 실행 결과·체크인 응답
- **출력**: 주간 진행 현황 + 코칭 피드백 + 신규 발견 후보 및 자산화 저장소 갱신(→ 4.4로 재투입) + 개인/팀 활용 대시보드
- **정책 연계**: Layer 구분(2.2) — 코칭 피드백도 Layer별 원칙 준수 (Layer 3 코칭은 판단 대행 금지). AI 만능주의 경계(2.3) — 트래킹 중 AI로 해결 부적합한 이슈 발견 시 구분 안내. 인사 불이익 자동화 금지(2.6) — 활용 대시보드는 감시·평가 도구로 쓰지 않음
- **오픈 이슈**: 체크인 발송 주기·형식, 막힘 감지 임계값, 평가 결과의 4.2 재진단 반영 방식, 자산화 저장소를 외부 툴(예: Notion 등)에 위임하는 옵션 제공 여부 미정 (제공 시에도 세부 연동 방식은 스프린트 문서에서 다룸)

### 4.6 Bottom-up 리포트 생성

- **목적**: 팀 단위 성과를 조직 내 확산시키기 위한 설득용 리포트 생성 (경영진/동료 대상)
- **포함 지표 (예시)**: AI 활용률 변화(예: 도입 전 대비 몇 %), 절감 시간(주간/월간/연환산), 가장 효과 좋았던 업무 사례, 새로 쌓인 자산화 저장소 항목 수(위키/프롬프트), 팀 참여율
- **산출물 형태**: 1페이지 요약 카드(성공 사례 중심) — 상사에게 캡처·공유하기 쉬운 형태로 구성. 상세 버전은 문서(추후 export 형식은 스프린트 문서에서 결정)
- **입력**: 트래킹 결과(4.5) + 평가 지표(4.4) + 자산화 저장소 현황(4.5)
- **출력**: Bottom-up 리포트 (1페이지 요약 + 상세, 성과 요약과 재현 가능한 방법 포함)
- **정책 연계**: 미검증 통계 확정적 제시 금지(2.6) — 모든 수치는 자체 측정값이거나 출처 명시된 인용만 사용. 실행 가능성 최우선(2.4) — 리포트도 뭉뚱그린 전략 언어 지양
- **오픈 이슈**: 상세 리포트 포맷, 확산 대상(팀 내 vs 팀 밖) 미정
