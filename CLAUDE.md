# 문서 작업 방식 (Context)

이 repo의 문서 체계와 작업 규칙을 기록한다. 새로 합류하거나 새 대화를 시작할 때 가장 먼저 읽는다.

## 문서 구조

| 문서 | 역할 |
|---|---|
| `SPEC.md` | 기획 의도, 기능, **합의된 정책**. 이 repo에서 작업할 때 가장 먼저 참조하고 의사결정의 기준으로 삼는다. |
| 스프린트 문서 | 기능 단위 기술/구현 방법. `SPEC.md`에는 기술/구현 방법을 담지 않는다. |

### 스프린트 문서 네이밍/구조

- 파일명: `SPRINT{n}_CONTRACT.md` (해당 스프린트에서 여러 기능을 동시에 구현할 때 기능 간 공통 인터페이스 계약), `SPRINT{n}_FEATURE{번호}_{기능명}.md` (기능별 상세 구현 문서)
- 여러 기능이 서로 데이터를 주고받으며 동시에 개발될 때는 반드시 `CONTRACT` 문서를 먼저 만들어 인터페이스(입출력 스키마, 역할 분담)를 고정하고, 각 기능 담당자는 자신의 `FEATURE` 문서에서 세부 구현을 채운다.
- 계약이 바뀌면 `CONTRACT` 문서를 갱신하고 관련된 모든 기능 담당자에게 공유한다 (한쪽 문서에서만 몰래 바꾸지 않는다).
- 현재 진행 중: **스프린트1** — 기능 3(BP 리서치 엔진) · 기능 4(맞춤 로드맵 생성) 구현. `SPRINT1_CONTRACT.md`, `SPRINT1_FEATURE3_BP_RESEARCH.md`, `SPRINT1_FEATURE4_ROADMAP_GENERATOR.md` 참고.

`SPEC.md`는 한 번 쓰고 끝나는 문서가 아니라, 새로운 세부 기능이나 정책이 합의될 때마다 계속 추가·수정되는 살아있는 기준 문서다.

## 업데이트 규칙

1. **입력**: "기능 고도화 제안서"가 들어온다 (기능별 대안 제안).
2. **비교**: 기능별로 제안서의 대안들을 비교한다.
3. **합의**: 사용자와 상호작용하며 대안 중 채택안을 결정한다.
4. **반영**: `SPEC.md`를 다음 세 가지 방식 중 하나로 업데이트한다.
   - **추가**: 기존 스펙에 없던 내용을 신규로 추가
   - **대안 중 결정**: 여러 대안 중 하나를 확정하고 나머지는 기각 사유와 함께 기록
   - **옵션**: 아직 확정하지 않고 선택지로만 남겨둠 (추후 결정)
5. **기록**: `SPEC.md` 하단 변경 이력에 날짜·변경 내용·근거를 남긴다.

## 기능 범위 (고정)

기능은 아래 6개로 **고정**한다. 세부 정책/구현이 바뀌어도 이 6개 skeleton 구조 자체는 변경하지 않는다.

1. 온보딩 인터뷰
2. AX 성숙도 진단 및 목표 설정
3. BP 리서치 엔진
4. 맞춤 로드맵 + 평가 지표 생성 (목표 → task 분해, 평가지표)
5. 로드맵 실행 트래킹 / 주간 코칭 및 평가
6. Bottom-up 리포트 생성

## 하지 않는 것

- `SPEC.md`에 기술 스택, API, 프롬프트 엔지니어링 등 구현 세부사항을 적지 않는다. → 스프린트 문서로 분리.
- 기능 개수를 늘리거나 줄이지 않는다. 세부 기능은 6개 기능 내부의 하위 항목으로 편입한다.
